## [Unreleased] — 2026-03-07

### Added ✨

- **feat(faq-queries): Full customer FAQ submission + admin management system**

  **Module — `src/modules/faq-queries/`**

  | File | Purpose |
  |---|---|
  | `models/faq-query.ts` | Medusa DML data model |
  | `service.ts` | `MedusaService`-based CRUD service |
  | `index.ts` | Module registration, exports `FAQ_QUERY_MODULE` constant |
  | `migrations/Migration20260307170333.ts` | Auto-generated DB migration (applied ✅) |

  **`models/faq-query.ts` — schema:**
  ```ts
  model.define("faq_query", {
    id:             model.id().primaryKey(),
    customer_name:  model.text(),
    customer_email: model.text(),
    subject:        model.text(),
    question:       model.text(),
    answer:         model.text().nullable(),
    status:         model.enum(["pending", "answered"]).default("pending"),
    answered_at:    model.dateTime().nullable(),
  })
  ```
  Full soft-delete support via Medusa DML (`created_at`, `updated_at`, `deleted_at` auto-injected).

  **`service.ts`** — wraps `MedusaService({ FaqQuery })`. Auto-provides:
  `createFaqQueries`, `listFaqQueries`, `listAndCountFaqQueries`, `retrieveFaqQuery`, `updateFaqQueries`, `deleteFaqQueries` — no hand-written CRUD needed.

  **`index.ts`:**
  ```ts
  export const FAQ_QUERY_MODULE = "faq_query"
  export default Module(FAQ_QUERY_MODULE, { service: FaqQueryModuleService })
  ```

  **`Migration20260307170333.ts`** — generated by `yarn medusa db:generate --modules faq_query`. Creates `faq_query` table with all 11 columns, `CHECK (status IN ('pending','answered'))` constraint, `IDX_faq_query_deleted_at` index. Applied via `yarn medusa db:migrate`. **Verified live in PostgreSQL** — all 11 columns confirmed: `id · customer_name · customer_email · subject · question · answer · status · answered_at · created_at · updated_at · deleted_at`.

- **feat(api/store): `POST /store/faq-queries` — public customer submission endpoint**

  File: `src/api/store/faq-queries/route.ts`

  | Aspect | Detail |
  |---|---|
  | Auth | Public — no JWT required |
  | Method | `POST` |
  | Validation | `customer_name` 2–100 · `customer_email` RFC regex · `subject` 3–150 · `question` 10–2000 chars |
  | Response 201 | `{ faq_query: { id, subject, status, created_at } }` — PII fields (`customer_email`) deliberately excluded |
  | Error 400 | `{ error: "Validation failed", details: { field: "message" } }` |
  | Rate limited | 5 requests/hour/IP (see Security section) |

- **feat(api/admin): Admin FAQ query management endpoints**

  Files: `src/api/admin/faq-queries/route.ts` · `src/api/admin/faq-queries/[id]/route.ts`

  | Route | Method | Purpose |
  |---|---|---|
  | `/admin/faq-queries` | `GET` | List all queries. Query params: `status` (`pending`/`answered`/`all`, default `all`), `limit` (default 50), `offset` (default 0) |
  | `/admin/faq-queries/:id` | `GET` | Retrieve single query (all fields including PII — admin only) |
  | `/admin/faq-queries/:id` | `PATCH` | Post or update answer. Sets `status: "answered"` and `answered_at: new Date()`. Answer validated 10–3000 chars. |
  | `/admin/faq-queries/:id` | `DELETE` | Delete query (soft-delete via Medusa service) |

  All admin routes are behind Medusa's built-in admin JWT middleware — no additional auth needed.

- **feat(admin-ui): `src/admin/routes/faq-queries/page.tsx` — "FAQ Queries" Medusa Admin sidebar page**

  Follows the exact same pattern as the existing `product-reviews` and `cod-fraud` custom admin routes in this codebase.

  **Sidebar registration:**
  ```ts
  export const config = defineRouteConfig({
    label: "FAQ Queries",
    icon: QuestionMarkCircle,   // from @medusajs/icons
  })
  export const handle = { breadcrumb: () => "FAQ Queries" }
  ```
  Placing `page.tsx` under `src/admin/routes/faq-queries/` is sufficient for the Medusa Admin Vite bundler to add the route to the sidebar automatically — no manual route registration needed.

  **UI features:**
  - Header with live pending-count badge (amber `bg-orange-100 text-orange-700`)
  - Status filter tabs: **All** / **Pending** (orange badge) / **Answered** (green badge)
  - Table columns: Customer (name + email), Subject, Date, Status badge, Actions
  - Click any row → expandable detail panel (full question text, existing answer with edit date, or inline answer form)
  - Answer form: `<Textarea>` (Medusa UI component) + character counter (`n / 3000`) + "Post answer" / "Update answer" / "Cancel"
  - Delete button per row
  - `useMutation` (custom hook wrapping `fetch`) for PATCH and DELETE with `toast.success` / `toast.error` Medusa toast feedback
  - Empty state: `QuestionMarkCircle` icon + "No queries yet" (all tab) / "No pending queries — all caught up! 🎉" (pending tab)

### Security 🔐

- **security(faq-queries): `faqQueryRateLimiter` middleware — 5 req/hour/IP**

  File: `src/api/middlewares.ts`

  Prevents spam and abuse of the public `POST /store/faq-queries` endpoint. Architecture mirrors the existing OTP and COD rate limiters already in this codebase:

  **Implementation — Redis Lua atomic counter (primary) + in-memory `Map` fallback:**
  ```ts
  // Redis path (atomic — no TOCTOU race):
  const script = `
    local cur = redis.call('INCR', KEYS[1])
    if cur == 1 then redis.call('EXPIRE', KEYS[1], 3600) end
    return cur
  `
  // Fallback (Redis unavailable):
  const entry = inMemoryFaqRateLimit.get(key) ?? { count: 0, resetAt: now + 3600_000 }
  ```

  Key format: `faq_rate:<ip>` — TTL 3600s (resets hourly). Limit: 5 requests before 429.

  Hourly `setInterval` in `middlewares.ts` prunes expired entries from `inMemoryFaqRateLimit` to prevent unbounded memory growth in long-running instances (same pattern as the COD in-memory limiter BUG-006 fix above).

  Route entry added to middleware matrix:
  ```ts
  { matcher: "/store/faq-queries", method: ["POST"], middlewares: [faqQueryRateLimiter] }
  ```

## [Unreleased] — 2026-02-26

### Security 🔐

### Security 🔐
- **BUG-006**: Fixed memory leak in COD in-memory OTP rate limiter. Expired entries are now cleaned up every hour, preventing unbounded growth if Redis recovers and fallback is not triggered again.
- **COD Fraud Detection System** — end-to-end feature tracking COD cancellation behaviour:
  - `src/lib/util/cod-fraud.ts` — shared metadata reader/writer, block message formatter
  - `src/subscribers/cod-fraud-tracker.ts` — `order.canceled` subscriber: increments `cod_cancellations` strike on customer-initiated COD cancellations; sets `cod_blocked: true` after threshold; records a queued admin notification
  - `src/subscribers/cod-unlock-tracker.ts` — `order.placed` subscriber: decrements `cod_online_orders_needed` counter and auto-unlocks COD when the customer completes enough online-payment orders
  - `src/api/store/cod/eligibility/route.ts` — `GET /store/cod/eligibility` — COD availability check + block reason for checkout UI
  - `src/api/store/cod/cancellation-risk/route.ts` — `GET /store/cod/cancellation-risk/:id` — pre-cancellation risk warning for the customer
  - `src/api/store/cod/notifications/route.ts` — `GET /store/cod/notifications` — admin-queued toast messages, auto-cleared on read
  - `src/api/admin/custom/cod-fraud/route.ts` — `GET/POST /admin/custom/cod-fraud` — admin override: view stats, manually block/unblock customers
  - `src/admin/routes/cod-fraud/page.tsx` — Medusa Admin UI panel for COD fraud management
  - `src/admin/widgets/cod-fraud-widget.tsx` — order-detail widget showing COD risk badge
  - `src/modules/cod-payment/service.ts` — `initiatePayment()` now checks `cod_blocked` metadata and throws `NOT_ALLOWED` before creating a payment session
  - `src/api/store/orders/[id]/cancel/route.ts` — `POST /store/orders/:id/cancel` — customer self-cancellation endpoint; stamps `cancelled_by: "customer"` on the order so the fraud tracker can distinguish customer-initiated vs. admin-initiated cancellations
- **`src/lib/util/retry.ts`** — generic `retryWithBackoff(fn, { attempts, baseDelayMs, factor })` utility with configurable exponential backoff
- **`src/jobs/flush-search-buffer.ts`** — scheduled job (every 30 s) that drains a Redis set `search:buffer:pending` and calls the sync workflow once for all buffered product IDs as a single batch

### Fixed 🐛
- **Shiprocket transient failures**: `src/modules/shiprocket-fulfillment/service.ts` — all three Shiprocket API steps (createOrder, generateAWB, schedulePickup) are now wrapped with `retryWithBackoff` (3 attempts, 2s/4s/8s). Expired auth tokens are refreshed transparently by `getHeaders()` on each attempt.
- **Shiprocket missing AWB guard**: `src/services/shiprocket.ts` — `generateAWB()` now throws `MedusaError(UNEXPECTED_STATE)` if the response body contains no `awb_code`, instead of silently returning `undefined` and creating an untrackable fulfillment record.
- **Admin navigation (SPA routing)**: `src/admin/lib/razorpay-shared.tsx` and `src/admin/routes/razorpay/page.tsx` — replaced all `window.location.href = ...` calls with React Router `useNavigate()`. Prevents full-page reloads and loss of Admin SPA state on hotkey navigation.

### Changed 🔄
- **`src/subscribers/product-sync.ts`** — refactored from direct workflow call to Redis buffer (`SADD search:buffer:pending <id>`). 100 rapid product updates now trigger 1 sync batch instead of 100 individual workflow executions.
- **`src/api/hooks/razorpay/route.ts`** — reduced to ~40 LOC (HMAC verify + enqueue). All event routing logic (refund email, payment logging) moved to `src/modules/razorpay-queue/processor.ts`.
- **`medusa-config.ts`** — registered `razorpay-queue` as a custom module. Documented BullMQ Worker architecture, Redis requirement, and per-event idempotency in inline comments.
- **`package.json`** — added `bullmq@5.70.1` runtime dependency.
- Security tooling: `eslint-plugin-security`, `eslint-plugin-no-secrets`, `secretlint`, `@typescript-eslint` devDependencies
- `.eslintrc-security.js` — security-focused ESLint config with rules for Math.random, eval, ReDoS, timing-unsafe comparisons, deprecated crypto, secret detection
- `.secretlintrc.json` — secretlint config using recommended preset
- `BUGS.md` — living vulnerability tracker with 12 initial findings from security audit
- `CHANGELOG.md` — this file
- `SECURITY_ANALYSIS.md` — full security analysis report
- `src/scripts/create-meilisearch-scoped-key.ts` — one-time setup script for Meilisearch scoped key (BUG-001 fix)
